Client Layer — Three Entry Points

PredictX reaches users through three distinct client interfaces, each optimized for a different engagement pattern. Together they form the hybrid UX approved in the design sprint: a Snap-style discovery feed in-app, WhatsApp as the distribution channel, and a PWA for frictionless web access.

1. React Native Mobile App (Primary)

The mobile app is the core experience — a dark-themed, Snap-style vertical feed with inline YES/NO action buttons. Built with React Native 0.76 and TypeScript for shared iOS/Android code, with Reanimated 3 powering buttery 60fps swipe animations even on $100 Tecno/Infinix phones.

Edge & API Gateway Layer

The gateway layer sits between users and backend services. It handles authentication, rate limiting, request routing, and real-time WebSocket connections at scale. Designed to serve millions of concurrent users across Africa, South Asia, Southeast Asia, and Europe.

CloudFlare Edge Network

CloudFlare provides the global CDN with edge PoPs close to every target market. Edge Workers run lightweight logic (geofencing checks, A/B routing, static response caching) at the edge, reducing round-trip latency from 200ms+ to under 50ms for cached responses.

  • DDoS Protection: Enterprise-tier DDoS mitigation — critical for a betting platform that attracts attacks
  • WAF Rules: Custom rules blocking known bot patterns, credential stuffing, and API abuse
  • Image Optimization: Auto-resize and compress market card images for low-bandwidth users
  • Edge Caching: Market listings, leaderboards, and static content cached at 300+ global PoPs

API Gateway — Kong

Kong Gateway handles all API traffic with plugins for authentication, rate limiting, request transformation, and observability. It routes requests to the appropriate microservice via service discovery.

WebSocket Gateway — Elixir/Phoenix

Real-time is non-negotiable for a prediction platform. The WebSocket gateway, built on Elixir/Phoenix Channels, handles millions of concurrent persistent connections for live odds, chat, feed updates, and notification streaming. Elixir's BEAM VM is purpose-built for this — each connection is a lightweight process consuming ~2KB of memory.

  • Live Odds: Odds changes broadcast to all subscribers of a market within 50ms
  • Market Chat: 500+ concurrent users per market room, message fan-out via Redis Pub/Sub
  • Activity Feed: Friend prediction activity streamed in real-time to social feed
  • Presence: Online/typing indicators for chat rooms and Tribe activity

Core Microservices

PredictX's backend is a microservices architecture, with each service owning its domain, database, and API. Services communicate asynchronously via Kafka events for loose coupling, with synchronous gRPC for latency-sensitive paths. All services are containerized and deployed to Kubernetes.

Auth & Identity Service (Go)

Handles all authentication, user identity, and compliance-related identity verification. Phone OTP is the primary auth method (no email required), with social login as a convenience layer. Tiered KYC integrates with Sumsub for document verification.

Market Service (Go)

The central service managing prediction market lifecycle — from creation through trading to resolution. Each market has a category, resolution criteria, time bounds, and configurable parameters.

  • Market States: Draft → Active → Suspended → Resolving → Settled → Archived
  • Resolution: Automated (API data feeds for sports scores, weather, stock prices) + Manual judges for subjective markets + AI validation layer
  • Categories: Sports, Entertainment, Politics, Weather, Finance, Viral, Local — configurable per country
  • User-Generated Markets: Users propose markets → community votes → AI validates resolution criteria → admin approves

Wallet & Payment Service (Go)

Manages dual wallet system: virtual currency (free-to-play, India) and real money (Wave 1 markets). Each currency has its own ledger with double-entry bookkeeping. Payment integrations are region-specific.

Social & Tribes Service (Node.js/TypeScript)

Powers all social features — profiles, activity feeds, friend connections, Tribes (persistent social groups), challenges, and WhatsApp sharing. Uses Neo4j for the social graph (friend-of-friend queries, Tribe recommendations) and Redis Streams for the real-time activity feed.

Gamification Engine (Go)

Drives daily engagement through streaks, XP, badges, levels, and leaderboards. Redis Sorted Sets power real-time leaderboards (regional + global, weekly/monthly/all-time) that update within milliseconds of each prediction. Cron jobs handle daily streak evaluation, weekly Tribe competitions, and seasonal ranking resets.

Notification Service (Node.js)

The Pulse engine — contextual, smart notifications that bring predictions to users at relevant moments. Not spam. Integrates with Firebase Cloud Messaging (Android), APNs (iOS), and SMS fallback. Kafka consumer processes notification events from all services and applies frequency capping, quiet hours, and user preference filters before delivery.

Content & Localization Service (Node.js)

Manages the i18n framework (react-i18next on client, Crowdin for translation management), local market curation per country, and content pipelines that surface culturally relevant markets. Supports English, Yoruba, Pidgin English, Swahili, Hindi, Tagalog, and Spanish at launch with a target of adding new languages in under 48 hours.

Matching Engine — The Heart of PredictX

The matching engine is a Rust-based, high-performance order matching system that combines a Central Limit Order Book (CLOB) with an Automated Market Maker (AMM). This hybrid approach provides deep liquidity via AMM for low-volume markets while enabling tight spreads via CLOB for high-volume ones.

Performance Targets

Sub-10ms order matching latency · 10,000+ orders/second throughput · 99.99% uptime · Zero data loss (event-sourced)

Why Rust?

The matching engine is the one component where latency and correctness are existential. Rust provides memory safety without garbage collection pauses, zero-cost abstractions, and predictable sub-microsecond latencies. The LMAX Disruptor pattern (single-writer, mechanical sympathy) is implemented natively in Rust's async runtime (Tokio) for lock-free, sequential order processing.

Risk & Settlement (Rust)

The settlement engine runs as a separate Rust service, consuming trade events from Kafka. It manages position tracking (open positions per user per market), P&L calculations, and when a market resolves, it distributes payouts atomically using a two-phase commit with the Wallet Service's PostgreSQL database.

  • Fraud Detection: Real-time pattern detection for wash trading, collusion, and abnormal bet sizing
  • Responsible Gambling: Enforces deposit limits ($50 default), cool-off periods, self-exclusion, and loss alerts
  • Market Manipulation: Monitors for coordinated price movements across related markets

AI & Intelligence Layer

AI is woven throughout PredictX — not as a bolt-on feature but as core infrastructure. Four distinct AI systems power recommendations, market resolution, trend monitoring, and natural language understanding for the chat bot.

1. AI Recommendation Engine

Personalizes the feed, suggests markets, and ranks content for each user. Combines collaborative filtering (users like you predicted X), content-based signals (user's category preferences, past predictions), and contextual features (time of day, location, live events). Runs as a Python service with a feature store backed by Redis and ClickHouse.

2. Market Resolution Agent

Automates the resolution of prediction markets by combining structured data APIs with AI validation. Sports scores come from live data feeds (ESPN, API-Football), weather from OpenWeatherMap, financial data from market APIs. For subjective or complex markets, Claude API acts as a reasoning agent that evaluates evidence and recommends resolution, with human judges as the final arbiter.

  • Auto-Resolution Pipeline: Data feed triggers → AI validates data quality → Resolution proposed → 15-min dispute window → Settlement
  • Appeal System: Users flag disputed resolutions → AI reviews evidence → Escalate to human panel if confidence < 95%
  • Cron Scheduling: Celery Beat checks all active markets every 60 seconds for resolution triggers

3. Event Hijacking Engine

Monitors global trends in real-time and auto-creates prediction markets around viral events within 2 hours. Ingests signals from Twitter/X trending topics, Google Trends, news APIs, and sports fixtures. Apache Flink processes the stream, Claude API generates market questions and resolution criteria, and an admin review queue approves before publishing.

Example Flow

Burna Boy announces Coachella headline slot on Twitter → Flink detects spike → Claude generates "Will Burna Boy's Coachella set exceed 100K live viewers?" → Admin approves → Market live within 90 minutes of the tweet.

4. WhatsApp NLU Engine

Powers the conversational interface for the WhatsApp and Telegram bots. Uses Claude Haiku for fast intent classification (predict, check portfolio, deposit, help) and entity extraction (team names, market categories, amounts) across 7+ languages. spaCy handles tokenization and language detection for routing to the appropriate model.

Data & Storage Layer

PredictX's data layer is purpose-built for the dual demands of a financial system (ACID transactions, audit trails, zero data loss) and a social platform (real-time feeds, search, analytics). Each storage technology is chosen for its specific strength.

PostgreSQL 16 — Primary Database

The source of truth for all transactional data: users, markets, orders, trades, wallets, and compliance records. Deployed as a Citus-sharded cluster for horizontal scaling.

  • Sharding Strategy: Shard by user_id for wallet/order tables, by market_id for market/trade tables
  • Connection Pooling: pgBouncer with transaction-mode pooling — handles 10K+ concurrent connections
  • Replication: Synchronous replication to a standby in the same AZ for zero data loss; async replicas in other regions for read scaling
  • Audit: All financial mutations logged to an append-only audit table with pgcrypto checksums

Redis Cluster 7 — Cache & Real-time

Redis serves multiple roles: session store (JWT + refresh tokens), cache (market data, odds, user profiles), real-time leaderboards (Sorted Sets), rate limiting (sliding window counters), and pub/sub for WebSocket fan-out.

  • Cluster Mode: 6-node cluster (3 masters + 3 replicas) for high availability
  • Memory: ~50GB cluster, with eviction policy for cache data (allkeys-lru) and persistence for critical data (leaderboards)
  • Pub/Sub: Market-specific channels for odds updates fan out to all WebSocket instances

Neo4j — Social Graph

Stores friend relationships, Tribe memberships, and social interactions. Graph queries power "friends who predicted" social proof, Tribe recommendations, and influencer identification.

ClickHouse — Analytics Warehouse

Columnar analytics database for all event-level data: predictions, page views, funnel events, revenue, cohort analysis. Ingests from Kafka via ClickHouse Kafka Engine. Materialized views pre-aggregate hourly/daily metrics. Powers Grafana dashboards and the admin analytics panel.

Elasticsearch — Search

Full-text search across markets (multi-language), user profiles, and market comments. Powers the "Search" functionality in the app and the WhatsApp bot's fuzzy market matching.

Apache Kafka — Event Bus

The central nervous system connecting all services. Every significant event (order placed, trade matched, market resolved, user registered, payment processed) is published to Kafka. This enables event sourcing for the matching engine, real-time analytics, notification triggers, and full system auditability.

TopicProducersConsumers

Object Storage — AWS S3

Stores user avatars, market card images, KYC documents (encrypted, separate bucket with restricted IAM), matching engine state snapshots, and analytics exports.

Kubernetes (EKS) Architecture

  • Node Pools: General-purpose (m6i.xlarge) for API services, compute-optimized (c6i.2xlarge) for matching engine, memory-optimized (r6i.xlarge) for Redis/caching

Key System Flows

End-to-end flow of the most critical user journeys through the system architecture.

Flow 1: User Makes a Prediction (Swipe-to-Bet)

1
Client (React Native): User swipes right (YES) on a market card. App sends POST /v1/orders with market_id, side (YES), amount.
2
CloudFlare Edge: Request passes WAF rules, routed to nearest API Gateway.
3
Kong Gateway: Validates JWT, rate-limit check, routes to Order Service.
4
Order Service (Go): Validates balance from Wallet Service (gRPC), checks responsible gambling limits, checks market is active. Publishes order to Kafka orders.placed.
5
Matching Engine (Rust): Consumes order. Checks CLOB for matching counter-orders. If no match, fills against AMM at current LMSR price. Publishes trade to Kafka trades.matched.
6
Settlement (Rust): Consumes trade. Debits buyer's wallet, credits position. Updates market volume. Atomic transaction with PostgreSQL.
7
WebSocket Gateway (Elixir): Consumes trade. Broadcasts updated odds to all market subscribers. Sends trade confirmation to the user's WebSocket connection.
8
Social Service: Publishes activity to user's friends' feeds. Gamification engine awards XP.
9
Client: Receives WebSocket confirmation. Plays celebration animation. Generates shareable WhatsApp card.
End-to-end latency target: < 200ms

From swipe gesture to confirmation animation on the client. Matching engine contributes < 10ms. Network + API overhead is the dominant factor, reduced by edge proximity.

Flow 2: WhatsApp Bot Prediction

1
User sends "Arsenal win?" in WhatsApp → Meta webhook fires to Bot Service.
2
NLU Engine (Claude Haiku): Classifies intent = "predict", extracts entity = "Arsenal". Searches Elasticsearch for relevant markets.
3
Bot Service: Returns rich media template with market details, current odds, and YES/NO interactive buttons.
4
User taps YES → Webhook fires again. Bot Service creates order via Order Service (same flow as app). Sends confirmation card back to WhatsApp.

Flow 3: Market Resolution (Automated)

1
Celery Beat (cron): Every 60s, checks all active markets for resolution triggers (time-based, API data match).
2
Resolution Agent (Python): Fetches outcome data from external API (e.g., football score from API-Football). Validates data quality.
3
AI Validation (Claude): Cross-references multiple data sources. Confirms resolution with high confidence.
4
Market Service: Transitions market to "Resolving" state. 15-minute dispute window opens.
5
Settlement Engine: After dispute window, calculates payouts. Winners' wallets credited atomically. Losers' positions marked as settled.
6
Notifications: Push notifications sent to all market participants with result + P&L. Gamification engine updates streaks and accuracy scores.

Flow 4: Real-Money Deposit (Nigeria — Paystack)

1
Client: User taps "Add Funds" → selects bank transfer → enters NGN amount.
2
Payment Service (Go): Initiates Paystack transaction. Returns checkout URL.
3
Paystack: User completes payment (bank transfer or card). Paystack webhook fires on success.
4
Payment Service: Validates webhook signature. Credits user's NGN wallet via double-entry ledger write to PostgreSQL.
5
KYC Check: If deposit exceeds 10K NGN and user is Tier 1, triggers Sumsub KYC upgrade flow (BVN verification).
6
Notification: Push notification confirms deposit. Wallet balance updates in real-time via WebSocket.
